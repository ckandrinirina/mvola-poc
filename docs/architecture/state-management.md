# State Management

> Companion to [data-flow.md](data-flow.md) and [components.md](components.md) — covers the in-memory store layer that backs the wallet, transaction, and game features.

## Overview

The PoC has **no database**. All domain state lives in module-level `Map` objects inside `src/lib/store/`. This mirrors the existing OAuth token cache in `src/lib/mvola/auth.ts` — survive process lifetime, reset on server restart, zero new runtime dependencies.

All mutations happen **server-side only**. UI components observe state by polling the relevant internal API routes.

## Stores

| File | Map | Key | Purpose |
|------|-----|-----|---------|
| `src/lib/store/wallets.ts` | `Map<string, WalletState>` | `msisdn` | Current balance per player |
| `src/lib/store/transactions.ts` | `Map<string, TransactionRecord>` | `localTxId` | Every deposit and cash-out, with reconciliation state |
| `src/lib/store/games.ts` | `Map<string, GameSession>` | `sessionId` | Every coin-flip round played |

`transactions.ts` additionally maintains a secondary index `Map<string, string>` mapping `correlationId → localTxId` for fast webhook/status lookups.

## Schemas

### `WalletState`
```ts
interface WalletState {
  msisdn: string;          // primary key
  balance: number;         // integer Ariary, always ≥ 0
  createdAt: number;       // ms since epoch
  updatedAt: number;       // ms since epoch
}
```

### `TransactionRecord`
```ts
type TransactionDirection = "deposit" | "withdraw";
type TransactionStatus    = "pending" | "completed" | "failed";

interface TransactionRecord {
  localTxId:       string;                 // primary key, generated by this server
  correlationId:   string;                 // MVola serverCorrelationId (secondary index)
  msisdn:          string;                 // player's MVola number
  direction:       TransactionDirection;
  amount:          number;                 // integer Ariary
  status:          TransactionStatus;
  walletSettled:   boolean;                // see "Idempotency" below
  mvolaReference?: string;                 // MVola's transactionReference, set on completion
  createdAt:       number;
  updatedAt:       number;
}
```

### `GameSession`
```ts
type GameChoice  = "heads" | "tails";
type GameResult  = "win" | "loss";

interface GameSession {
  sessionId:     string;        // primary key
  msisdn:        string;
  bet:           number;        // integer Ariary
  choice:        GameChoice;
  outcome:       GameChoice;
  result:        GameResult;    // derived: outcome === choice ? "win" : "loss"
  delta:         number;        // +bet on win, −bet on loss
  balanceAfter:  number;        // wallet balance immediately after this round
  playedAt:      number;
}
```

## Accessor contracts

Stores expose **only typed functions** — the underlying `Map` is never exported. This keeps call-sites easy to grep and future-proofs the layer against swapping in a real database.

### `wallets.ts`
| Function | Behaviour |
|----------|-----------|
| `getWallet(msisdn)` | Returns the `WalletState` or `undefined` |
| `ensureWallet(msisdn)` | Returns an existing wallet or creates a zero-balance one |
| `creditWallet(msisdn, amount)` | `ensureWallet` + `balance += amount` + bump `updatedAt` |
| `debitWallet(msisdn, amount)` | Throws `InsufficientFundsError` if `balance < amount`; otherwise `balance -= amount` |
| `resetAll()` | Test-only: clears the wallet map |

### `transactions.ts`
| Function | Behaviour |
|----------|-----------|
| `createTransaction(input)` | Generates `localTxId`, inserts into map, indexes by `correlationId` |
| `getTransactionByCorrelationId(correlationId)` | Secondary-index lookup |
| `updateTransactionStatus(localTxId, status, patch?)` | Updates `status`, optional `mvolaReference` / `walletSettled`, bumps `updatedAt` |
| `listTransactionsByMsisdn(msisdn)` | Filters and sorts by `createdAt` desc |
| `resetAll()` | Test-only |

### `games.ts`
| Function | Behaviour |
|----------|-----------|
| `recordGameSession(input)` | Generates `sessionId`, inserts into map |
| `listGameSessionsByMsisdn(msisdn)` | Filters and sorts by `playedAt` desc |
| `resetAll()` | Test-only |

## Idempotency

Both the **status poll** and the **webhook callback** can deliver the same terminal outcome (`completed` or `failed`) for a given transaction. Without care, the wallet could be credited or refunded twice.

The `walletSettled` flag on every `TransactionRecord` prevents that:

| Direction | Wallet mutation trigger | Initial `walletSettled` | After reconciliation |
|-----------|--------------------------|--------------------------|----------------------|
| **Deposit** | Credit on first `completed`. No wallet change on `failed`. | `false` at request time | `true` as soon as a terminal status is observed |
| **Withdraw** | Debit at request time (reserve). Refund on `failed`. | `true` at request time (already reserved) | stays `true` on `completed`; flipped to `false` on `failed` then credited back |

Every reconciliation call-site (status route + callback route) must:

1. Look up the local record by `correlationId`
2. **Read** the current `status` and `walletSettled`
3. If the incoming status is terminal AND the transition is new (status is `pending` OR `walletSettled` has not yet been applied for this direction), apply the wallet side-effect
4. Update the record's `status` and `walletSettled` in a single call

Duplicate deliveries become no-ops because the second call observes that the record is already in its final state.

## Invariants

- `WalletState.balance >= 0` always — enforced by `debitWallet` throwing `InsufficientFundsError`
- Every amount field (wallet balance, transaction amount, game bet) is an **integer Ariary**; no floats
- `TransactionRecord.direction` never changes after creation
- `TransactionRecord.correlationId` is unique per record; the secondary index enforces this
- A `GameSession` is only ever inserted, never updated — the game is append-only
- All timestamps are `ms since epoch` (from `Date.now()`); there is no timezone handling needed for the PoC

## Concurrency

The PoC assumes a **single Node.js server process**. Node's single-threaded event loop serialises all async operations, so read-modify-write sequences like "check balance, then debit" are safe within a single API route handler **as long as the handler does not `await` between the check and the mutation**.

- The store modules therefore expose *compound* operations (`creditWallet`, `debitWallet`) rather than separate getters and setters — the atomic step happens synchronously inside a single function call.
- The deposit / withdraw / coin-flip route handlers follow the same discipline: they perform balance validation and mutation before any `await` that could yield the event loop.

If this PoC is ever scaled horizontally (multiple processes or Edge runtime), the in-memory model breaks and must be replaced with a transactional store — see the **Out of Scope** section of [overview.md](overview.md).

## Reset behaviour

- **Server restart (`npm run dev` relaunch):** wallets, transactions, and game sessions are all wiped. The OAuth token cache is also wiped (existing behaviour).
- **Programmatic reset:** each store exposes a `resetAll()` helper for tests only; do not call from production code paths.
- **Manual inspection / reset in dev:** there is no "clear state" admin route in the PoC. If one is desired later, it should sit behind a dev-only flag and call each store's `resetAll()`.

## Debug tips

- `GET /api/wallet/:msisdn/balance` is the cheapest way to see the current wallet state
- `GET /api/wallet/:msisdn/history` shows the full reconciliation trail: each transaction's `walletSettled` flag is exposed, so you can verify that the status route or the callback has run
- If a deposit looks stuck in `pending`, check both the status route and the webhook route — at least one should eventually flip the record to `completed`
- The coin-flip route is fully synchronous: if `GET /api/wallet/:msisdn/balance` does not change after a `200` response, the deposit/cash-out path has a bug, not a timing issue
