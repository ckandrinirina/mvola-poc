# Ship — Commit, PR & Issue Management

Commit your changes, optionally create a PR, and update any linked GitHub Issues.
Automatically detects story context to link everything together.

**CRITICAL RULE: Never mention Claude, Claude Code, AI, or any AI assistant in
commits, PRs, comments, or any git/GitHub output. No co-author tags, no
"generated by", no AI references of any kind.**

---

## INPUT

`$ARGUMENTS` is an optional path to a story file for context.

**If `$ARGUMENTS` is provided:** Read the story to extract issue links and context.

**If `$ARGUMENTS` is empty:** Check for story context from the current branch name
or recent git activity. If none found, proceed as a standalone commit.

---

## PHASE 0: BRANCH CHECK

**Goal:** Ensure work is on a feature branch — never commit directly to `main` or `develop`.

### 0.1 Check Current Branch

```bash
git branch --show-current
```

### 0.2 Act on Result

**If already on a feature branch** (e.g. `story/01-03-*`, `fix/02-01-*`):
→ Branch is correct. Skip to Phase 1.

**If on `main`, `develop`, or any protected branch:**
→ STOP before staging or committing anything.
→ Propose a branch name derived from the story context:
  - Story: `story/[EE]-[SS]-[short-slug]`
  - Bug fix: `fix/[EE]-[SS]-[short-slug]`
  - Standalone: `[type]/[short-slug]`

Present:
```
You are on `main`. All story work should go through a PR.

Suggested branch: story/01-03-server-setup

A) CREATE — Create this branch now and continue
B) RENAME — Use a different name
C) SKIP — Commit to main anyway (not recommended)
```

If A or B → create and switch to the branch:
```bash
git checkout -b [branch-name]
```

If C → warn once more, then proceed to Phase 1.

**This check applies to standalone commits too** — if on a protected branch,
always offer to create a feature branch before proceeding.

## PHASE 1: GATHER CONTEXT

**Goal:** Understand what we're committing and what it's linked to.

### 1.1 Check Git State

```bash
git status
git diff --stat
git diff --staged --stat
git log --oneline -5
```

If nothing to commit (clean working tree + no staged changes):
→ "Nothing to commit. Working tree is clean." → STOP

### 1.2 Detect Story Context

Try to find the linked story in this order:

1. **From `$ARGUMENTS`:** If a story path was provided, read it
2. **From branch name:** Parse `story/[EE]-[SS]-*` or `fix/[EE]-[SS]-*` branch patterns
3. **From recent files:** Check if any modified files match a story's "Files to Create/Modify"

If a story is found, extract:
- Story title, epic, ID (EE-SS)
- Status (should be DONE or IN PROGRESS)
- Acceptance criteria (for PR description)
- Implementation Summary or Bug Resolution (if present)
- **Files Touched** section (if present)

### 1.3 Detect Linked GitHub Issues

If a story is found, check for linked GitHub Issues:

1. **Read the story file** for any issue references (`#123`, `GH-123`, etc.)
2. **Search GitHub Issues** matching the story title:
   ```bash
   gh issue list --label "story" --state open --json number,title \
     | jq '.[] | select(.title | contains("[EE-SS]"))'
   ```
3. **Check the parent epic** for its issue number:
   ```bash
   gh issue list --label "epic" --state open --json number,title \
     | jq '.[] | select(.title | contains("Epic [NN]"))'
   ```

Store found issues:
- `story_issue`: The GitHub Issue for this specific story
- `epic_issue`: The GitHub Issue for the parent epic

### 1.4 Read Commit Style

Check recent commits to match the repo's commit message style:
```bash
git log --oneline -10
```

## PHASE 2: PREPARE COMMIT

**Goal:** Stage files and craft a clean commit message.

### 2.1 Stage Files

Review what's changed and stage appropriately:

```bash
git status
```

**Auto-stage rules:**
- Stage all modified/new source files related to the story
- Stage all new/modified test files
- Stage story file updates (status, summary, files touched)
- **NEVER stage:** `.env`, credentials, secrets, `.DS_Store`, IDE configs

Present what will be staged:
```
## Files to Stage

**Source files:**
- [file list]

**Test files:**
- [file list]

**Documentation:**
- [story file updates]

**Excluded (not staging):**
- [any excluded files and why]

Stage these files? YES / ADJUST
```

### 2.2 Craft Commit Message

**Format:** Follow conventional commits, adapted to the context.

**For story implementation (`/build`):**
```
feat([epic-slug]): [story title]

Implements story [EE]-[SS]: [brief description]

- [Key change 1]
- [Key change 2]
- [Key change 3]

[count] acceptance criteria met
[count] tests passing

[If GitHub issue exists:]
Closes #[story_issue_number]
```

**For bug fix (`/fix`):**
```
fix([epic-slug]): [bug description]

Fixes bug in story [EE]-[SS]: [root cause]

- [What was changed]
- Added regression test: [test name]

[If GitHub issue exists:]
Closes #[story_issue_number]
```

**For standalone commit (no story):**
```
[type]([scope]): [description]

[body — what and why]

[footer — issue references if any]
```

Where `type` is: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `style`, `perf`

### 2.3 Confirm Commit

```
## Commit Preview

**Branch:** [current branch]
**Files:** [count] staged
**Message:**
[full commit message]

[If issue linked:]
**Linked issues:** #[number] (will be closed on merge)

Commit? YES / EDIT MESSAGE / ABORT
```

If EDIT MESSAGE → let user modify, then re-confirm.

## PHASE 3: COMMIT

### 3.1 Execute Commit

```bash
git add [specific files]
git commit -m "[message]"
```

Use a HEREDOC for multi-line messages:
```bash
git commit -m "$(cat <<'EOF'
feat(foundation): setup WebSocket server

Implements story 01-03: WebSocket gateway with MessagePack serialization

- Created ws handler with connection management
- Added MessagePack serialization/deserialization
- Integrated with existing gRPC client

5 acceptance criteria met
12 tests passing

Closes #42
EOF
)"
```

### 3.2 Verify Commit

```bash
git log --oneline -1
git show --stat HEAD
```

Present:
```
## Committed

**Hash:** [short hash]
**Branch:** [branch name]
**Files:** [count]
**Message:** [first line]
```

## PHASE 4: CREATE PR (Optional)

### 4.1 Ask About PR

```
Create a pull request? 

A) YES — Create PR now
B) NO — Just the commit, no PR
C) LATER — Push branch, I'll create PR manually
```

If NO → skip to Phase 5.
If LATER → push and skip to Phase 5.

### 4.2 Determine PR Target

```
Target branch for PR?

A) main (default)
B) develop
C) [other — specify]
```

### 4.3 Push Branch

```bash
git push -u origin [branch-name]
```

### 4.4 Craft PR

**PR Title:** Same as commit first line (keep under 70 chars).

**PR Body:**

For story implementation:
```markdown
## Summary
- Implements story [EE]-[SS]: [title]
- Epic: [epic title]

## Changes
- [Key change 1]
- [Key change 2]
- [Key change 3]

## Acceptance Criteria
- [x] [Criterion 1]
- [x] [Criterion 2]
- [x] [Criterion 3]

## Testing
- [count] automated tests added/modified
- All tests passing
- Manual testing completed

[If GitHub issue exists:]
Closes #[story_issue_number]
```

For bug fix:
```markdown
## Summary
- Fixes bug in story [EE]-[SS]: [title]
- Root cause: [explanation]

## Changes
- [What was fixed]
- Added regression test: [test name]

## Testing
- Reproduction test now passes
- No regressions detected
- [count] total tests passing

[If GitHub issue exists:]
Closes #[story_issue_number]
```

### 4.5 Create PR

```bash
gh pr create \
  --title "[title]" \
  --base [target-branch] \
  --body "$(cat <<'EOF'
[PR body]
EOF
)"
```

Present:
```
## PR Created

**URL:** [PR URL]
**Title:** [title]
**Target:** [base branch]
**Linked issues:** #[numbers]
```

## PHASE 5: UPDATE GITHUB ISSUES

**Goal:** Update linked GitHub Issues with the commit/PR information.

### 5.1 Update Story Issue

If a linked story issue was found:

**If PR was created:**
Add a comment to the story issue:
```bash
gh issue comment [story_issue_number] --body "$(cat <<'EOF'
Implementation complete. PR: #[pr_number]

Changes:
- [Key change 1]
- [Key change 2]

All acceptance criteria met. [count] tests passing.
EOF
)"
```

**If commit only (no PR) and branch is main/develop:**
Close the issue:
```bash
gh issue close [story_issue_number] --comment "$(cat <<'EOF'
Implemented in [commit_hash].

Changes:
- [Key change 1]
- [Key change 2]

All acceptance criteria met. [count] tests passing.
EOF
)"
```

### 5.2 Update Epic Issue

If a linked epic issue was found:

Check the epic's story checklist and mark the completed story:
1. Read the epic issue body:
   ```bash
   gh issue view [epic_issue_number] --json body -q .body
   ```
2. Find the line `- [ ] #[story_issue_number]` and replace with `- [x] #[story_issue_number]`
3. Update the epic issue body:
   ```bash
   gh issue edit [epic_issue_number] --body "[updated body]"
   ```

This updates the epic's task list progress in GitHub.

### 5.3 Add Labels

If the story issue exists, add a status label:
```bash
gh issue edit [story_issue_number] --add-label "status/done"
```

For bug fixes:
```bash
gh issue edit [story_issue_number] --add-label "has-bugfix"
```

## STANDALONE MODE (No Story)

When no story context is detected, the skill works as a clean commit+PR tool:

1. Show `git diff --stat` and `git status`
2. Ask user what type of change this is (feat/fix/refactor/etc.)
3. Ask for a brief description
4. Craft conventional commit message
5. Commit, optionally PR
6. No issue updates (no story to link)

---

## IMPORTANT GUIDELINES

### ABSOLUTELY NO AI REFERENCES
- **NEVER** add "Co-Authored-By: Claude" or any AI co-author tag
- **NEVER** mention "Claude Code", "Claude", "AI", "generated", "assisted" in commits
- **NEVER** add "Generated with Claude Code" or similar to PR descriptions
- **NEVER** add AI references in issue comments
- Commit messages and PRs should read as if written by a human developer
- This rule is absolute and cannot be overridden

### Commit Messages Must Be Clean
- Follow conventional commits format
- Keep first line under 70 characters
- Body explains WHAT and WHY, not HOW
- Include issue references (`Closes #123`) when applicable
- No emoji in commit messages unless the repo convention uses them

### Stage Selectively
- Never use `git add -A` or `git add .`
- Stage specific files by name
- Never stage secrets, credentials, or environment files
- Review what's being staged before committing

### Issue Updates Are Careful
- Only close issues when the work is actually complete
- Use `Closes #X` in commit/PR to let GitHub auto-close on merge
- Comment on issues to provide progress updates
- Update epic checklists to reflect story completion

### Branch Naming
- Story implementation: `story/[EE]-[SS]-[slug]`
- Bug fix: `fix/[EE]-[SS]-[slug]`
- Standalone: `[type]/[slug]` (e.g., `feat/add-logging`, `fix/null-check`)
- **Never commit directly to `main` or `develop`** — Phase 0 enforces this

### Reusability
- Works with any project using the `tasks/` story format
- Also works standalone without any story context
- No project-specific references
